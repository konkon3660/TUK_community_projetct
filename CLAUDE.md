**이 파일은 "사양을 어떻게 지킬 것인가"에 대한 규칙**입니다. 사양 자체는 여기 옮겨 적지 마세요.

- 무엇을 만들어야 하는가 → `documents/02_requirements.md` (확정 사양)
- 화면 컨벤션 → `documents/06_gui.md`
- 진행 현황 → `documents/08_status.md`

## 0. 왜 이 파일이 생겼나

2026-07-23 실사용 점검에서 아래 결함이 한꺼번에 나왔습니다.

| # | 증상 | 원인 유형 |
|---|---|---|
| 1 | 홈에 게시판 6종이 그대로 나열됨 (대분류 없음) | 화면 설계 미확정 |
| 2 | 공지의 "대상 학과"가 자유 입력 텍스트 | 선택형 UI 미적용 |
| 3 | 큰 이미지가 화면 밖으로 넘침 | 렌더링 시 스케일 누락 |

| 5 | 공동구매 글에서 연결된 채팅방에 못 들어감 | 화면 연결 누락 |

| 9 | 관리자가 민원 글을 수정/삭제할 수 있음 | 사양 자체가 잘못됨 |

**공통 원인 두 가지**입니다. 새 작업을 시작하기 전에 반드시 읽으세요.

### (A) 요구사항 문서에 적혀 있는데 구현이 빠졌다

4·6·7·8은 전부 `02_requirements.md`에 **문장으로 명시되어 있었습니다.**
그런데 `08_status.md`에는 "GUI 🟩 완료 — 화면 15개 전부"라고 적혀 있었습니다.
화면 클래스가 존재하는 것과 그 화면의 요구사항이 충족된 것은 다릅니다.

> **규칙 A-1.** 화면/기능을 건드리기 전에 `02_requirements.md`에서 그 기능에 해당하는 절을
> 열고, **문장 하나 = 체크 항목 하나**로 훑으세요. 표의 행, 불릿 하나까지 전부 체크 대상입니다.
>
> **규칙 A-2.** "완료"라고 보고하기 전에, 그 절의 각 문장에 대해 *어느 클래스의 어느 메서드가
> 그걸 하는지* 말할 수 있어야 합니다. 말할 수 없으면 완료가 아닙니다.
>
> **규칙 A-3.** 모델 계층에 메서드가 있다고 완료가 아닙니다. `ChatRoom.setNickname()`은
> 처음부터 있었지만 **아무 화면도 그것을 부르지 않아서** 닉네임 기능이 없었습니다.
> 새 모델 메서드/필드를 만들거나 발견하면 **호출하는 화면이 있는지 grep 해서 확인**하세요.
>
> **규칙 A-4.** `08_status.md`의 🟩 표시를 신뢰하지 마세요. 코드가 정답입니다.
> 상태를 갱신할 때는 "구현했다"가 아니라 "요구사항 §X의 항목 N개를 전부 확인했다"로 적습니다.

### (B) 화면을 띄우는 것까지만 하고 실제로 클릭해보지 않았다

1·3·5는 실행해서 한 번만 눌러봤으면 즉시 보였을 문제입니다.

> **규칙 B-1.** GUI를 수정했으면 **컴파일만으로 끝내지 말고 실제로 실행해서 그 경로를 클릭**하세요.
> ```powershell
> javac -encoding UTF-8 -d out (Get-ChildItem -Recurse -Filter *.java | % FullName)
> java -cp out server.CT.ServerMain      # 터미널 1
> java -cp out client.GUI.MainFrame      # 터미널 2 (서버가 먼저 켜져 있어야 시작됨)
> ```
> 계정은 `08_status.md §4-2` 표 참고 (관리자 `2026000001`/`admin01`, 학생 `2026591007`/`pass01`).
>
> **규칙 B-2.** 새로 만든 화면은 `MainFrame.main()` 등록만으로는 도달할 수 없습니다.
> **① 등록 ② 그 화면으로 가는 버튼 ③ 그 화면에서 돌아오는 버튼** 세 개가 다 있어야 완료입니다.
> 시간표 입력 화면(#4)이 바로 ②가 없어서 못 쓰는 상태였습니다.
>
> **규칙 B-3.** 데이터가 커지면(긴 제목, 큰 이미지, 항목 100개) 레이아웃이 깨지는지 확인하세요.
> 기본 데이터로만 테스트하면 #3 같은 문제를 못 봅니다.

## 1. 화면 구조 규칙

### 1-1. 홈은 2단계 네비게이션이다 (#1)

`HomePanel`에는 **대분류 4개만** 둡니다:

```
게시판  |  민원  |  채팅  |  할 거 추천
```

"게시판"을 누르면 그 다음 화면에서 **자유 / 공동구매 / 학과 / 기숙사 (+ 공지사항)** 를 고릅니다.
게시판 6종 버튼을 `HomePanel`에 직접 늘어놓지 마세요.

- 중간 단계는 별도 화면(`boardMenu` 등)으로 만들고 `MainFrame.main()`에 등록합니다.
- 뒤로가기는 `게시판 목록 → 게시판 대분류 → 홈` 순으로 한 단계씩 돌아가야 합니다.
  `PostListPanel`의 `backTarget`이 `"home"`으로 고정되어 있으면 한 단계를 건너뛰게 되므로
  중간 화면을 넣을 때 이 값을 함께 고치세요 (`backTarget`은 `"admin"`도 받습니다 — `06_gui.md §2`).
- 관리자(`AdminPanel`)는 이 구조와 **무관**합니다. 관리자가 학생 화면으로 가는 경로를
  만들지 마세요 (`06_gui.md §2`, 기획 확정 사항).

### 1-2. 값이 정해져 있는 입력은 자유 입력이 아니라 선택형으로 (#2)

`JTextField`에 사용자가 학과명을 직접 타이핑하면 오타 하나로 기능이 조용히 죽습니다
(학과 게시판의 `boardKey`가 **곧 학과명 문자열**이라 `"AI소프트웨어학과"`와 `"AI소프트웨어 학과"`는
다른 게시판입니다).

**학과를 입력받는 곳은 전부 `JComboBox`(드롭다운)** 여야 합니다:

| 화면 | 필드 | 현재 상태 |
|---|---|---|
| `NoticePostEditorPanel` | 대상 학과 (복수 선택, 비우면 전체 공지) | 🔴 쉼표 구분 텍스트 |
| `RegisterPanel` | 학과 | 🔴 텍스트 |
| `UserEditPanel` | 학과 | 🔴 텍스트 |
| `ChatRoomCreatePanel` | 학과 제한 (복수 선택) | 🔴 쉼표 구분 텍스트 |

> **규칙.** 학과 목록의 **단일 출처(single source of truth)를 하나 만들고** 위 네 곳과
> 서버가 모두 그것을 참조하게 하세요. 지금은 `server/board/DataStore.registerBoards()`에만
> 하드코딩되어 있어 클라이언트가 알 방법이 없습니다.
> `model/`에 상수 클래스를 두면 서버·클라이언트가 같이 씁니다 (`model/`은 양쪽 공용 —
> `01_overview.md §3`). **화면마다 학과 목록을 따로 하드코딩하는 것은 금지**입니다.
> 새 학과가 생겼을 때 고칠 곳이 5군데가 되면 반드시 하나를 빠뜨립니다.
>
> 같은 원칙이 적용되는 다른 항목: 민원 1차/2차 카테고리, 공동구매 해시태그
> (`02_requirements.md §3.1` — "**미리 정해진 목록에서 고른다**"고 이미 명시되어 있습니다).

### 1-3. 이미지와 긴 콘텐츠는 화면 밖으로 나가면 안 된다 (#3)

`PostDetailPanel.loadImage()`는 원본 크기 그대로 `ImageIcon`을 붙이고 있어서 큰 이미지가
창을 밀어냅니다.

> **규칙.** 이미지를 표시할 때는 **원본 가로세로 비율을 유지한 채** 표시 영역에 맞춰
> 축소합니다. 확대는 하지 않습니다(작은 이미지는 원본 크기 그대로).
> - 축소 비율 = `min(1.0, 영역너비/원본너비, 영역높이/원본높이)` — 두 축 중 **작은 쪽**을
>   써야 비율이 유지됩니다. 가로만 맞추면 세로가 넘칩니다.
> - `Image.getScaledInstance(w, h, Image.SCALE_SMOOTH)`로 줄인 뒤 `ImageIcon`으로 감쌉니다.
> - 그래도 넘칠 수 있으므로 이미지 영역은 `JScrollPane` 안에 둡니다.
>
> **일반화:** 크기를 예측할 수 없는 것(이미지, 본문, 목록, 채팅 기록)은 전부 `JScrollPane`
> 안에 넣고, `setPreferredSize`로 고정 크기를 박지 마세요. 창 크기는 900×600이 기본이지만
> 사용자가 줄일 수 있습니다.

## 2. 기능별 재발 방지 규칙

### 2-1. 공동구매 글 ↔ 채팅방은 양방향으로 오갈 수 있어야 한다 (#5)

`GroupBuyPost.chatRoomId`로 방이 이미 연결되어 있고 `PostDetailPanel.findLinkedRoom()`이
그 방 객체를 이미 들고 있는데(참여 인원 표시용), **들어가는 버튼이 없습니다.**

> **규칙.** 공동구매 게시글 상세에는 "채팅방 들어가기" 버튼이 있어야 합니다.
> - 이미 참여자면 → `((ChatRoomPanel) getScreen("chatRoom")).open(room)` 후 `switchTo("chatRoom")`
> - 참여자가 아니면 → **가입 신청 절차를 먼저 탑니다.** 참여자가 아닌 채로 채팅방 화면에
>   들여보내면 서버가 `CHAT_SEND`를 거부해서 아무것도 못 합니다
>   (`ChatRoomListPanel.requestJoin()`과 **같은 흐름을 재사용**하세요. 복붙하지 말고 공통화).
> - `linkedRoom`이 `null`이면(채팅방 연동 전 예전 글) 버튼을 감춥니다.

### 2-2. 채팅방 검색 (#6) / 게시글 검색

`02_requirements.md §4.4`와 §2.2-6에 둘 다 명시되어 있습니다.

> **규칙.** 검색은 **새 `RequestType`을 만들지 말고** 이미 받아온 목록을 클라이언트에서
> 거르는 방식으로 구현합니다 (`CHATROOM_LIST` / `POST_LIST` 결과를 필터링).
> 서버 수정이 필요 없고, 프로토콜 표(`05_protocol.md`)도 안 건드립니다.
> `RequestType`에 상수를 추가해야만 하는 상황이면 **먼저 그 enum의 주석대로 문서부터** 고치세요.
>
> 같이 넣어야 하는 것: `§4.4`의 "가입한 채팅방 보기 — 공동구매 채팅방 / 일반 채팅방 탭 분리",
> `§3.1`의 해시태그 필터.

### 2-3. 민원 템플릿 (#7)

`02_requirements.md §3.5`: "**자주 묻는 질문 탭에 민원 템플릿을 제공**한다. 사용자가 템플릿을
불러와서 이어 쓸 수 있다."

> **규칙.** 템플릿은 **정적 로컬 데이터**입니다 — 서버 통신 없이 클라이언트가 파일에서 읽습니다
> (추천 3종이 `client/recommend_data/`를 읽는 것과 같은 방식).
> - 템플릿 1개 = 제목 + 1차 분류 + 2차 분류 + 본문 뼈대.
> - 템플릿을 고르면 `ComplaintPanel`의 제목/분류/내용 칸이 **채워지고, 사용자가 이어서 수정**할 수
>   있어야 합니다 (읽기 전용으로 보여주기만 하면 요구사항 미충족).
> - 파일 형식은 `model/DataFormat`의 구분자 규칙을 따르고, **UTF-8(BOM 없이)** 로 저장합니다
>   (`08_status.md §4-2`의 경고 참고 — BOM이 첫 필드에 섞여 파싱이 깨집니다).

### 2-4. 채팅방 입장 시 닉네임 설정 (#8)

`02_requirements.md §4.2`: "입장할 때 이 방에서 쓸 **프로필 이름(닉네임)** 을 설정할 수 있다."
`ChatRoom.setNickname()` / `getNickname()`은 이미 있고, `ChatRoomPanel`도 표시할 때
닉네임을 우선 사용합니다. **설정하는 UI와 그것을 서버에 저장하는 경로만 없습니다.**

> **규칙.** 닉네임은 **방마다 다릅니다**(`Map<userId, nickname>`). 전역 프로필이 아닙니다.
> - 입장 시 한 번 물어보고, 채팅방 화면 안에서 **언제든 다시 바꿀 수 있어야** 합니다.
> - 서버에 반영하는 경로가 없으면 새로 만들어야 합니다. 그 경우 `RequestType` enum 주석대로
>   **상수 추가 + `documents/05_protocol.md` 표 갱신을 같은 커밋에서** 하세요.
>   `ClientHandler`에서 임의로 새 타입을 만들어 쓰면 안 됩니다.
> - 닉네임을 안 정했으면 학번을 그대로 보여줍니다 (`ChatRoomPanel`의 현재 동작 유지).
> - 닉네임에도 `DataFormat.encode()`가 적용되어야 합니다 (`ChatRoom`이 맵을 인코딩해서 저장 중).

### 2-5. 관리자는 남의 글을 수정할 수 없다 / 민원은 답변만 (#9)

이건 구현 실수가 아니라 **사양이 잘못된 경우**였습니다. **2026-07-23 기획 변경으로 사양 문서는
이미 새 규칙으로 고쳤고, 코드가 아직 예전 사양대로입니다** (`08_status.md §4-9`).

| | 예전 (현재 코드) | **새 사양** |
|---|---|---|
| 관리자가 남의 글 **수정** | 가능 | **불가** |
| 관리자가 남의 글 **삭제** | 가능 | 가능 |
| 관리자가 **민원** 수정/삭제 | 가능 | **둘 다 불가 — 답변(댓글)만** |

관리자가 남의 글 내용을 고칠 수 있으면 글이 기록으로서 의미가 없습니다. 부적절한 글에 대한
운영 행위는 "고쳐 쓰기"가 아니라 "내리기"입니다.

> **구현 시 주의:**
> 1. **`model/`의 `canXxx()`만 고칩니다.** `Post.canEdit()`에서 `isAdmin()` 예외를 빼고,
>    `ComplaintPost`에서 `canEdit`/`canDelete`를 오버라이드합니다.
> 2. **화면에서 `isAdmin()`을 직접 보고 분기하지 마세요.** `PostDetailPanel`은 이미
>    `post.canEdit(me)` 결과를 그대로 쓰므로 모델만 고치면 버튼이 저절로 맞습니다.
>    반대로 화면에서만 버튼을 숨기면 **서버는 그대로 뚫려 있습니다** (`06_gui.md §5-3`).
> 3. 서버(`ClientHandler.handlePostUpdate`, `AbstractBoard.removePost`)는 이미 같은
>    `canXxx()`를 부릅니다. **확인만 하고 새 분기를 추가하지 마세요.**
> 4. 권한 규칙을 또 바꾸게 되면 **사양 문서(02/04/06/07)와 코드를 같은 커밋에서** 고치세요.
>    이번 변경으로 고친 문서 목록이 `08_status.md §4-9`에 있습니다.

### 2-6. 시간표 입력 화면 (#4)

`02_requirements.md §5.1`: "**시간표를 직접 입력하는 화면을 따로 두고, 언제든 수정할 수 있게 한다.**"

현재 `ActivityRecommender.loadTimetable()`은 `client/recommend_data/time_table.dat`를
**읽기만** 합니다. 저장 메서드도, 입력 화면도, 그 화면으로 가는 버튼도 없습니다.

> 1. `ActivityRecommender`에 시간표 **저장** 메서드 (읽기 경로와 같은 파일, 같은 형식)

> 3. **`RecommendPanel`의 "할 거" 탭에서 그 화면으로 가는 버튼** — 지금 빠진 게 정확히 이것입니다.

## 3. 코드를 고칠 때 항상 지킬 것

1. **사양 문서와 코드는 같은 커밋에서 함께 움직입니다.** 사양이 바뀌면 `documents/02~07`을,
   진행 상황이 바뀌면 `08_status.md`만 고칩니다. 진행 상황을 사양 문서에 적지 마세요.
2. **`model/`은 서버·클라이언트 공용입니다.** 여기에 클라이언트 전용/서버 전용 코드를 넣지 마세요.
3. **화면 전환은 `open(...)` 먼저, `switchTo(...)` 나중** (`06_gui.md §3`).
   목록으로 돌아갈 때는 `PostListPanel.refresh()`를 부르지 않으면 방금 지운 글이 그대로 보입니다.
4. **실패 응답은 서버가 준 `getErrorMessage()`를 그대로 띄웁니다.** 클라이언트에서 메시지를
   새로 만들지 마세요 (서버가 한국어로 보냅니다).
5. **푸시(`onPush`)는 네트워크 스레드에서 옵니다.** Swing을 직접 건드리지 말고
   `SwingUtilities.invokeLater`로 감싸세요.
6. **사용자 입력값을 파일에 쓸 때는 `DataFormat.encode()`.** 제목의 `|` 하나, 본문의 줄바꿈
   하나로 다음 서버 기동이 깨집니다 (`04_data_model.md §1-1`).
7. **`javac` 에러 0개 + 코드 내 `TODO` 0개를 유지합니다.**
   ```powershell
   Get-ChildItem -Recurse -Filter *.java | Select-String "TODO: 구현 필요"
   ```
8. 프로젝트 최상위(`Java_summer_p`)에서 실행합니다. `server/data/...`를 상대 경로로 읽습니다.
