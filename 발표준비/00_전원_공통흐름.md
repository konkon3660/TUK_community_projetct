# 00. 전원 필독 — 5분 흐름도 (파트 상관없이 다 외우기)

> 교수가 "버튼 누르면 무슨 일이 일어나냐"고 물으면 **누구든** 이 한 흐름은 말할 수 있어야 함.
> 자기 파트만 알고 경계를 모르면 "협업 안 했다"로 보임.

## 로그인 버튼 하나로 보는 전체 흐름

```
[④ GUI]  LoginPanel: 로그인 버튼 클릭
          → Packet.request(LOGIN, LoginRequest(id,pw))   ← ③ 프로토콜이 만든 "봉투"
          → mainFrame.getConnection().sendRequest(요청)   ← ② 클라 네트워크

[② 클라넷] ServerConnection.sendRequest():
          요청에 붙은 requestId를 기억해두고, 소켓으로 객체를 보냄
          응답 올 때까지 블로킹(최대 10초)

        ────────────── 소켓 (TCP 5000) ──────────────

[① 서버넷] ClientHandler.run(): 소켓에서 Packet을 읽음
          → dispatch(): type이 LOGIN이라 handleLogin() 호출
          → DataStore에서 유저 찾고 비번 확인
          → Packet.success(요청, User) 로 응답 봉투 만들어 되돌려보냄

        ────────────── 소켓 ──────────────

[② 클라넷] readLoop가 응답 받음 → requestId로 짝 맞춰 sendRequest에 전달
[④ GUI]   response.getStatus()==OK 면 → 홈 화면으로 switchTo("home")
```

## 4개의 봉투(Packet) 종류 — 이것만은 전원 암기

| 종류 | 누가 만듦 | 언제 |
|---|---|---|
| `request` | 클라(GUI) | 서버에 뭔가 요청할 때 |
| `success` | 서버 | 요청 성공 응답 |
| `error` | 서버 | 요청 실패 (한국어 메시지 담김) |
| `push` | 서버 | **요청 없이** 서버가 먼저 보냄 (채팅·공지) |

- 요청/응답은 `requestId`로 짝을 맞춘다. **푸시는 requestId가 없다**(`isPush()`).
- 이 프로젝트의 자랑거리 = **푸시**(다른 사람 채팅이 실시간으로 내 화면에 뜸).

## 파트 지도 (기능별 담당 — 각자 화면·서버·저장을 세로로 다 걸침)

| 담당 | 커닝페이퍼 | 걸치는 범위 | 서버 통신 |
|---|---|---|---|
| **게시판** | `담당_게시판.md` | 게시판 화면 + Post/Board 모델 + 서버 handlePost/Comment/File | 있음 |
| **채팅방** | `담당_채팅방.md` | 채팅 화면 + ChatRoom/Chat 모델 + 서버 handleChat + **푸시** | 있음 |
| **할 거 추천** | `담당_할거추천.md` | 추천 화면 + recomment_system | **없음(로컬 파일)** |

> **핵심:** "게시판 담당"은 화면만이 아니라 **그 기능의 서버 처리·파일 저장까지** 다 자기 것.
> 자기 담당 파일 하나만 정독하면 됨. 아래 참고 파일은 통합/네트워크 담당이 보면 좋음.

### 기능 사이 경계 (교수가 "네 파트랑 옆 파트가 어떻게 이어지냐" 물으면)
- **게시판 ↔ 채팅방:** 공동구매 글을 쓰면 서버가 **같은 요청 안에서** 참여자 채팅방을 자동 생성해 연결한다.
- **추천은 독립적:** 다른 두 파트와 안 엮이고 서버도 안 탄다 — 이게 특징이자 답변거리.
