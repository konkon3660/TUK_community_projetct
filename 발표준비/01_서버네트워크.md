# ① 서버 네트워크 파트 (`server/CT/`)

**담당 파일:** `ServerMain.java`, `ClientHandler.java`, `SessionRegistry.java`

## 3줄 요약
- `ServerMain`은 포트 5000을 열고, **클라이언트가 접속할 때마다 스레드를 하나씩** 만들어 붙인다.
- `ClientHandler`가 그 스레드의 본체 — 소켓에서 요청(Packet)을 읽어 종류별로 처리한다.
- `SessionRegistry`는 "지금 누가 접속해 있나"를 관리해서, 채팅·공지 **푸시 대상**을 찾는다.

## 내 위치 (흐름에서)
클라가 보낸 요청이 소켓을 타고 오면 → **여기서 처리** → 응답을 되돌려보냄. 서버의 심장.

---

## 무조건 방어할 질문 5개 + 답

**Q1. 여러 명이 동시에 접속하면 어떻게 처리하나요?**
> `ServerMain`에서 `serverSocket.accept()`가 연결을 받을 때마다 `new Thread(new ClientHandler(...)).start()`로
> **접속 1개당 스레드 1개**를 붙입니다. 그래서 여러 명이 동시에 써도 서로 안 막힙니다.
>
> (후속: "왜 스레드풀 안 썼나?") → 소규모 교육용이라 단순하게 갔습니다. 접속이 많아지면 스레드가
> 너무 많아지니 `ExecutorService`(스레드풀)로 바꾸는 게 개선점입니다.

**Q2. 여러 스레드가 같은 게시글 데이터를 동시에 고치면 데이터가 깨지지 않나요?**
> `ClientHandler`에 `DATA_LOCK`이라는 static 객체 하나가 있고, 글쓰기·수정·삭제 같은
> **데이터를 바꾸는 요청은 전부 `synchronized(DATA_LOCK)`으로 감싸서** 한 번에 하나씩만 처리되게 했습니다.
> 단순 조회(목록 보기)는 락을 안 걸어서 빠릅니다.
>
> (후속: "왜 static?") → ClientHandler는 접속마다 새로 생겨서, 인스턴스 락이면 서로 다른 사람끼리는
> 안 막힙니다. 모두가 **하나의 락**을 공유해야 데이터를 안전하게 지킵니다.

**Q3. 채팅 메시지는 상대방 화면에 어떻게 실시간으로 뜨나요?**
> `SessionRegistry`가 `userId → ClientHandler` 맵을 들고 있어서, 채팅이 오면 그 방 참여자들의
> ClientHandler를 찾아 서버가 **먼저 push 패킷을 보냅니다**(`sendTo`). 요청-응답이 아니라
> 서버가 능동적으로 보내는 겁니다. 공지도 같은 방식(`sendToAll` + 조건 필터)입니다.

**Q4. 클라이언트가 보낸 값을 그대로 다 믿나요?**
> 아니요. 예를 들어 회원가입 때 클라가 "나 관리자야(isAdmin=true)"라고 보내도 서버가 **무조건
> false로 덮어씁니다**(`handleRegister`). 채팅방 id도 클라 값 무시하고 서버가 새로 채번하고,
> 글 쓸 때 작성자가 로그인한 사람과 다르면 거부합니다. **권한/식별 정보는 서버가 결정**합니다.

**Q5. 요청 처리 중 에러가 나면 어떻게 되나요?**
> `handleRequest`가 `try-catch`로 감싸고 있어서, 검증 실패 시 그냥 예외를 던지면
> 서버가 그걸 잡아 `Packet.error(요청, 예외메시지)`로 바꿔 보냅니다. 그래서 각 handle 메서드는
> `throw new IllegalStateException("한국어 메시지")`만 하면 되고, 그 메시지가 사용자 화면에 그대로 뜹니다.

---

## 이것만은 외우기
- **접속 1개 = 스레드 1개** (`ServerMain`의 `new Thread(...).start()`)
- **쓰기 요청은 `DATA_LOCK`으로 직렬화**, 조회는 안 함
- **푸시**: `SessionRegistry`로 대상 찾아 서버가 먼저 보냄
