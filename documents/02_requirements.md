## 1. 회원 / 인증

### 1.1 회원가입

입력 항목 (예시 값):

| 항목 | 예시 | 비고 |
|---|---|---|
| 학번(id) | `2026591007` | 앞 4자리 = 입학년도. 채팅방 가입 제한에 쓰임 |
| 학과 | `AI소프트웨어학과` | 학과 게시판 접근 권한의 기준 |
| 기숙사생 여부 | O / X | 기숙사 게시판·기숙사 공지의 기준 |
| 비밀번호 | `pass01` | |
| 관리자 여부 | X | **가입 화면에서 설정할 수 없다** (아래 규칙 참고) |

규칙:

1. **이미 등록된 학번으로는 재가입할 수 없다.** 서버가 거부한다.
2. **가입으로는 절대 관리자가 될 수 없다.** 클라이언트가 `admin=true`를 보내도 서버가
   무시하고 일반 유저로 만든다 (관리자 계정은 `users.dat`에 직접 넣어서 만든다).
3. 학번·비밀번호·학과는 필수 입력이다.

### 1.2 로그인과 화면 분기

로그인에 성공하면 **관리자와 학생은 완전히 다른 화면으로 간다.**

- 학생 → `home` (게시판/채팅/추천/민원)
- 관리자 → `admin` (공지 작성/게시글 관리/민원함/회원정보 수정)

관리자에게는 학생 화면으로 가는 경로가 아예 없고 로그아웃만 있습니다. 자세한 것은
[06_gui.md §2](06_gui.md)를 보세요.

### 1.3 회원정보 변경

전과, 기숙사 입·퇴실, 비밀번호 재설정, 계정 도용 신고 등 **회원정보 변경은 전부 관리자
문의**로 처리합니다. 사용자가 스스로 바꾸는 화면은 만들지 않습니다.

- 관리자는 학번으로 회원을 조회한 뒤 **학과 / 기숙사 여부 / 비밀번호**를 수정할 수 있습니다.
- **학번과 관리자 여부는 수정할 수 없습니다** (`User`에서 `final`).
- 현재는 관리자 계정이 하나라고 가정합니다 (부서별 분할은 향후 개선사항).

## 2. 게시판 공통

### 2.1 게시글

| 항목 | 설명 |
|---|---|
| 제목 | 작성자 입력 |
| 글쓴이 | 로그인한 사용자의 학번 (직접 입력 불가) |
| 내용 | 작성자 입력 |
| 파일 | 파일 경로 (최대 5MB) |
| 이미지 | 이미지 경로 |
| 댓글 | 댓글 객체 목록 |
| 작성 시각 | 자동 |

### 2.2 공통 규칙

1. **관리자는 모든 글을 수정/삭제할 수 있다.**
2. **사용자는 자신의 글만 수정/삭제할 수 있다.**
3. 수정/삭제 버튼은 위 조건에 맞지 않으면 **비활성화**한다 (`Post.canEdit` / `canDelete`).
4. 누구나 댓글을 달 수 있고, 댓글도 같은 규칙(관리자 또는 본인)으로 삭제한다.
5. 이미지/파일 첨부 가능.
6. 게시글 검색 기능.
7. **다른 사람 명의로 글이나 댓글을 쓸 수 없다** — 서버가 로그인 세션과 대조해서 거부한다.

### 2.3 게시판 종류와 접근 권한

| 게시판 | 조회 | 작성 | 비고 |
|---|---|---|---|
| **자유게시판** | 전체 | 전체 | |
| **공동구매게시판** | 전체 | 전체 | 채팅방 연결 + 해시태그 |
| **학과별 게시판** | 관리자 + 해당 학과 | 동일 | 학과마다 게시판 1개 |
| **기숙사 게시판** | 관리자 + 기숙사생 | 동일 | |
| **공지사항** | 전체(대상 필터링됨) | **관리자만** | |
| **민원** | **관리자** 또는 **본인 것만** | 전체 | 사용자↔관리자 1:1 |

## 3. 게시판별 세부 사양

### 3.1 공동구매 게시판

게시글에 추가되는 항목:

| 항목 | 설명 |
|---|---|
| 최대 인원 수 | 작성자가 설정 |
| 채팅방 | 자동으로 만들어져 연결되는 채팅방의 id |
| 해시태그 | 여러 개 가능. 예: `#기숙사`, `#대량구매`, `#음식`, `#생필품` |

- **현재 인원수 = 연결된 채팅방의 참여자 수.** 별도 카운터를 두지 않는다
  (`GroupBuyPost.getCurrentMemberCount(ChatRoom)`).
- 해시태그는 미리 정해진 목록에서 고르며, 해시태그별 필터링이 가능하다.

### 3.2 학과별 게시판

- 접근 제한 기준: `User.getDepartment()`와 게시판의 `restrictedDepartment`가 일치하는지.
- 관리자는 모든 학과 게시판에 들어갈 수 있다.
- 현재 등록된 학과: AI소프트웨어학과 / 컴퓨터공학과 / 물리학과.
  새 학과는 `DataStore.registerBoards()`에 한 줄 추가하면 된다.

### 3.3 기숙사 게시판

- 접근 제한 기준: `User.isDormitory()`.

### 3.4 공지사항

- 작성은 관리자(시설관리팀·학사운영팀)만.
- 공지를 쓸 때 **해당하는 학과(여러 개)** 와 **기숙사 공지 여부**를 지정할 수 있다.
- 대상 학과가 비어 있으면 **전체 공지**다.
- 사용자에게는 자기에게 해당하는 공지만 보인다 (`NoticePost.isVisibleTo`).
- 공지가 등록되면 **접속 중인 대상 사용자에게 실시간 푸시**된다 (`NOTICE_PUSH`).

### 3.5 민원

사용자가 관리자에게만 보내는 글입니다. 학과·학번이 함께 전달됩니다.

민원 글의 추가 항목:

| 항목 | 설명 |
|---|---|
| 문의 1차 카테고리 | |
| 문의 2차 카테고리 | |
| 이미지 첨부 | 이미지 경로 |
| 답변 여부 | 답변 대기(회색) / 답변 완료(파란색) |

동작:

- **관리자는 민원함에서 전체 민원을 본다.** 일반 사용자는 본인이 넣은 민원만 본다
  (같은 목록 화면을 재사용하되 서버가 다르게 응답한다).
- 관리자의 **답변은 댓글로 구현**한다. 관리자가 민원에 댓글을 달면 서버가 자동으로
  답변 완료 상태로 바꾼다.
- 목록을 클릭하면 문의 내용과 그 아래 답변이 보인다.
- **자주 묻는 질문 탭에 민원 템플릿을 제공**한다. 사용자가 템플릿을 불러와서 이어 쓸 수 있다.
  (템플릿은 정적 데이터 — 아직 미작성, [08_status.md](08_status.md) 참고)

## 4. 채팅

### 4.1 채팅방

| 항목 | 설명 |
|---|---|
| 방 id | 서버가 채번 (`001`, `002`, ...) |
| 방장 | 만든 사람의 학번 |
| 참여자 | 학번 목록 |
| 채팅 기록 | 채팅 객체 목록 |
| 최대 인원 | `-1`이면 무제한 |
| 입학년도 제한 | 학번 앞 4자리. 없으면 제한 없음 |
| 학과 제한 | 여러 개 가능. 비어 있으면 제한 없음 |
| 기숙사 여부 제한 | |
| 초대수락 여부 | true면 초대받은 사람은 위 제한을 무시하고 들어올 수 있음 |
| 방 안 프로필 이름 | 학번 → 닉네임 |

채팅 하나에는 **보낸 사람 / 보낸 시각 / 내용**이 들어갑니다.

### 4.2 가입 절차

```
사용자가 가입 신청 (+ 가입지원 메세지)
      ↓
방장이 신청 목록에서 확인
      ↓
수락 → 참여자에 추가 / 거절 → 신청 삭제
```

- 신청 시점에 **정원·입학년도·학과·기숙사 제한을 모두 검사**한다. 하나라도 걸리면 신청 자체가 거부된다.
- 이미 참여 중이면 신청할 수 없다.
- 입장할 때 이 방에서 쓸 **프로필 이름(닉네임)** 을 설정할 수 있다.

### 4.3 실시간 채팅

- 누가 메시지를 보내면 **같은 방의 다른 접속자에게 서버가 즉시 푸시**한다 (`CHAT_MESSAGE_PUSH`).
- 보낸 사람 본인은 응답으로 이미 알고 있으므로 푸시를 받지 않는다.
- 참여 중이 아닌 방에는 메시지를 보낼 수 없다.

### 4.4 채팅방 탐색

- 채팅방 이름으로 검색.
- 가입한 채팅방 보기 — 공동구매 채팅방 탭 / 일반 채팅방 탭으로 나눔.

## 5. 추천 기능 3종

세 기능 모두 **서버 통신이 없습니다.** 클라이언트가 `client/recommend_data/`의 파일을
읽기만 하며, 쓰기/수정/삭제가 없어 가장 단순한 부분입니다.

### 5.1 할 거 추천기

- 시간표를 직접 입력하는 화면을 따로 두고, 언제든 수정할 수 있게 한다.
- 교시 시각은 고정: **1교시 09:30부터 1시간 단위**로 9교시까지.
- 시간표를 미리 넣어두면 **오늘 날짜·현재 시각을 보고 공강 시간을 자동 인식**한다.
- 그 공강 시간 안에 끝낼 수 있는 할 거리를 추천한다.
- 할 거 데이터: **내용**(예: "노래방에 가기") + **대강 걸리는 시간(분)**(예: 30).
- 목록은 30~40개 정도 준비한다. 소요 시간은 49분 이하로 잡는다.
- 여러 명이 함께 노는 경우나 서로 공강이 다른 경우는 **이번 범위에서 고려하지 않는다.**

### 5.2 메뉴 추천기

- 오늘의 학식 + E동 정보를 그대로 출력한다 ("오늘 학식 뭐더라?"를 해결).
- 맛없을 때를 위해 **주변 식당 중 랜덤 2개**를 추천한다.
  대상 식당: 맘스터치, 신북경, 북경마라탕, 토마토김밥, 한스오믈렛, 산융, 수호식당,
  토스피아, 라온식당.
- **"메뉴 추천 다시 받기" 버튼**을 둔다 (누를 때마다 다시 랜덤).
- 메뉴 아래에 꿀팁 한 줄을 랜덤으로 붙인다.

### 5.3 책 추천기

- 학과별 추천 도서 중 랜덤 1권.
- 책 이름과 그 책이 어느 학과 것인지, 관련 내용이 함께 저장된다.

## 6. 관리자 기능 (07/23 확정)

관리자 화면(`admin`)에서 할 수 있는 것:

| 기능 | 설명 |
|---|---|
| **공지글 작성** | 대상 학과·기숙사 지정 포함 |
| **게시글 관리** | 게시판을 골라 들어가서 아무 글이나 삭제 |
| **민원함** | 전체 민원 조회 + 댓글로 답변 |
| **회원정보 수정** | 학번으로 조회 → 학과/기숙사/비밀번호 변경 |

관리자 기능에 새 모델 클래스는 필요 없습니다. 기존 `canEdit` / `canDelete` / `canWrite` /
`canAccess`를 호출해서 **버튼 활성화 여부만 판단**하면 됩니다.

## 7. 향후 개선사항 (이번 범위 밖)

이번 학기 제출 범위에는 넣지 않되, 발표 때 "다음 단계" 로 언급할 항목들입니다.

1. 복수전공 고려
2. 학교 도서관 책 추천을 **한국공학대학교 중앙도서관 사이트 크롤링**으로 실제 데이터화
3. 관리자를 **담당 부서별로 분할** (지금은 계정 1개 가정)
4. 설치형이 아닌 **모바일/웹**으로 만들어 접근성 향상
5. 할 거 추천기에 **AI 적용** — 여러 명일 때, 공강 시간이 서로 다를 때까지 세분화·자동화
6. **룰베이스 알고리즘으로 죽은 채팅방 자동 정리**
7. 채팅에서 **이미지·파일 공유**
